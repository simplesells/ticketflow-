## Context

当前系统无身份识别：首页三个 Tab 按钮自由切换角色，前端硬编码处理人（张工/李工/王工），后端完全不校验操作者身份。MVP 要求"最小角色登录 / 身份区分"，需要给系统加上第一层身份识别。

约束：不引入完整账号体系（无注册、无 OAuth、无密码加密），保持 SQLite 单表结构。

## Goals / Non-Goals

**Goals:**
- 用户名+密码登录，登录后系统知道你是谁、你是什么角色
- 登录后自动进入对应工作台，不再需要手动切换角色
- 后端 API 按角色拦截非法操作（提交者不能分派，调度者不能创建等）
- 三种角色登录后看到不同首页

**Non-Goals:**
- 不实现注册功能（用户预置即可）
- 不引入 JWT / session / cookie 机制（用最简单的请求头传身份）
- 不加密密码（MVP 阶段，明文即可）
- 不改数据库 schema（用户数据硬编码在代码中）
- 不做"提交者只能看自己的工单"（那是后续 Change 的权限细化）
- 不做 history operator 实名、完成者列表 API、登出等（留给下一个 Change `login-polish`）
- 不做 status 端点的细粒度角色→流转映射（MVP 阶段，三种角色都能调 status，流转规则表本身做合法性校验已足够）

## Decisions

| # | 决定 | 理由 | 备选方案 |
|---|------|------|----------|
| 1 | 用户数据硬编码在后端数组 | 不需要数据库迁移，改动最小，符合"不要完整账号体系" | SQLite users 表 → 过度设计 |
| 2 | 登录后返回 userId，前端存 localStorage + 每次请求带 `x-user-id` header | 最简单，无需 session/jwt，前后端解耦 | JWT → 过度；Cookie-session → 需要额外库 |
| 3 | 后端用中间件 `requireRole(roles[])` 拦截 | 一个中间件覆盖所有路由，不重复代码 | 每个路由自己判断 → 重复 |
| 4 | 前端 LoginPage 作为路由守卫根节点 | 未登录自动跳转 `/login`，不暴露工作台 | 每个组件自己判断 → 重复 |
| 5 | 登录后按角色只显示对应工作台首页 | 提交者不再看到调度者/完成者 Tab | 保留 Tab 但灰掉 → 半吊子 |
| 6 | 不改数据库，现有 `submitter` 列存提交者名，`assignee` 存处理人名 | 已足够，无需新表 | 加 users 外键 → 破坏现有 Demo 数据 |
| 7 | status 端点三种角色都放行 | Explore 发现 DispatcherView 也调 status（退回提交者、拒绝），流转规则表本身限制 from→to 合法性已足够 | 按角色限制 from→to 映射 → MVP 过度 |
| 8 | 创建工单时 submitter 自动取 `req.user.displayName`，前端去掉提交人输入框 | 登录的核心价值就是"系统知道是谁提交的"。不能让用户手填，否则登录白做 | 前端继续手填 → 登录失去意义 |
| 9 | ResolverView 初始值用登录用户 displayName | 与 #8 同类：完成者登录后应默认看到自己的工单，不是张工的。按钮保留不删（留给 B） | 继续保持 RESOLVERS[0] → 登录后体验不对 |

## Risks / Trade-offs

- [风险] 明文密码 → 仅 MVP 阶段，交付时说明这是演示简化
- [风险] x-user-id 可伪造 → 仅 MVP 阶段，下一个 Change 可以加 token 校验
- [风险] 删掉角色切换 Tab 后，Demo 的"三视角自由切换"演示能力丢失 → 用多开浏览器或无痕窗口替代演示
