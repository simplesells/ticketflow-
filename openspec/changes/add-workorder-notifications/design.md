## Context

现有系统在评论、分派、状态变更时不通知相关人员。用户需要手动刷新才能看到新评论或状态变化。通知是独立实体，需要在触发点（评论创建、状态变更、分派）自动插入记录。

## Goals / Non-Goals

**Goals:**
- 评论发表时通知工单的提交者和处理人（发评论者除外）
- 状态变更时通知工单的提交者和处理人（操作者除外）
- 分派时通知被分派的处理人
- 前端导航栏显示未读通知数徽标
- 通知列表弹窗展示，点击跳转工单并标记已读

**Non-Goals:**
- 不实现实时推送（WebSocket/SSE），前端手动点击铃铛刷新
- 不发送邮件/短信/外部通知
- 不支持通知偏好设置（哪些类型通知可配置）
- 不持久化"全部已读"之外的通知状态管理

## Decisions

1. **通知在业务代码里同步创建，不用事件总线**
   - 项目规模小，直接调用 `createNotification()` 比引入事件机制简单
   - 在 comments.ts / workorders.ts 的对应路由里加 3-5 行代码即可

2. **接收人规则**
   - 评论通知 → 提交者 + 处理人（排除发评论者自己）
   - 状态变更通知 → 提交者 + 处理人（排除操作者自己）
   - 分派通知 → 仅被分派的处理人

3. **前端通知入口放在全局导航栏（App.tsx 顶部）**
   - 不在各角色工作台里重复放
   - 登录后统一可见，点击铃铛展开下拉列表

4. **未读数通过独立 API `GET /api/notifications/unread-count` 获取**
   - 前端定期刷新铃铛数字即可
   - 也可放在全局导航的 useEffect 里，路由切换时刷新

## Risks / Trade-offs

- [耦合] 通知创建逻辑散落在业务路由里 → 项目规模小，可接受；后续可抽成 service
- [性能] 无分页 → MVP 阶段通知量小，暂不处理
- [通知创建失败不回滚业务] → 通知失败记录日志但不阻断主操作
